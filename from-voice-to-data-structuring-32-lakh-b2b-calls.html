Cost duplication: The same audio file triggered multiple expensive LLM calls - one per team

Inconsistent accuracy: Different prompts meant different extraction quality. One team's "quantity" field might capture buyer requirements, another might capture MOQ, and a third might capture seller capacity

Schema chaos: No standard definition of fields. Is "price" with GST or without? Is "quantity" what the buyer wants or what the seller can supply?

Maintenance burden: Each team needed separate approvals, separate optimization efforts, and separate debugging

When you're processing millions of calls, these inefficiencies compound fast.

Building a Single Source of Truth ðŸŽ¯

The solution was conceptually simple: extract all required data in one model call and store it in a centralized database that everyone can use.

But simple doesn't mean easy. We needed to:

  1. Understand what every team actually needed
  2. Design a schema that could serve multiple use cases
  3. Build a prompt that accurately extracts 60+ fields
  4. Create infrastructure that could handle the scale
  5. Ensure the accuracy matched or exceeded existing pipelines

Step 1: Requirement Consolidation

We gathered input from every team using PNS (Phone Network Service) call data. What fields do you need? How do you define them? What's the acceptable accuracy?

Then came the hard work of deduplication and standardization. Multiple teams wanted "price" - but they meant different things. We needed unified definitions that worked for everyone.

The result: 60+ standardized fields grouped into logical categories:

  • Product and category details
  • Specifications (machine type, material, capacity)
  • Price details (value, currency, GST inclusion)
  • Buyer requirements
  • Seller information
  • Quantities and MOQs

Step 2: Schema Design

With 32 lakh calls per month and 60+ fields per call, database design mattered. We needed a structure that could:

  • Support current use cases and future requirements
  • Allow efficient querying for analytics
  • Maintain relationships between related data
  • Scale without performance degradation

The schema groups related fields into logical tables with clear relationships, making it easy for downstream teams to query what they need.

Step 3: The Extraction Model

Model choice: Gemini 2.5 Flash

Key decision: We use structured output extraction directly from audio - no transcript generation step. The model listens to the call and outputs data in a fixed JSON format.

Why this matters: Structured output dramatically reduces hallucinations. Instead of asking the model to generate free-form text that we then parse, we define the exact schema we want. The model fills in the fields or marks them as null if information isn't present.

Step 4: Prompt Engineering for Accuracy

Getting one field right is easy. Getting 60+ fields consistently accurate across diverse B2B conversations is hard.

We went through multiple audit cycles, refining the prompt to handle edge cases:

MOQ vs. buyer quantity: Is the number mentioned what the buyer wants or the minimum order quantity?

Capacity vs. quantity: When a seller says "I can supply 1000 units," is that total capacity or the quantity for this order?

Category identification: Mapping natural language product descriptions to standardized categories

Current accuracy: ~94% for product, specification, quantity, and price extraction. User details (names, locations, contact info) are at 80% - still being improved.

Teams using the data confirmed this matches or exceeds the accuracy of their previous independent pipelines.

Centralized architecture: one extraction pipeline serving multiple downstream use cases
